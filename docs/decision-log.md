# Decision Log — Rilo Analysis Rebuild

> 目的：把 Bos 已拍板的产品 / UI / 交互 / 工程原则固化，避免执行中反复漂移。

## 2026-03-13 — Round 0 / 方向冻结

### 产品方向
- 本轮主线不是“寄养模块专项优化”，而是**全局测算工作台重构**。
- 目标是：在不破坏现有测算能力的前提下，完成**视觉重构 + 信息架构重构 + 耦合度下降**。
- 当前所有可调参数都重要，**不允许因“主线聚焦”而默认折叠**。
- 敏感度分析页面继续保持“全局敏感度分析”定位，不改成局部功能页。

### 交互与输入规则
- 营业天数：**可调**。
- 入住率：**0–100%**，保留 **1 位小数**。
- 入住率控件：**滑杆 + 数字输入**。
- 入住率超界处理：自动夹到 **0** 或 **100**。
- UI 层面默认使用**中文术语**；缩写只在**公式 / 计算表达 / 代码层**出现。

### 页面与信息架构
- 定义说明（Definitions）：**右侧抽屉为主**，用一个轻量按钮触发。
- 定义说明不应抢主视觉空间，不做重型常驻区。
- 参数配置页面在调参时必须展示**基础计算结果**，增强测算直观性。
- 概览页面不走“只剩极少 KPI”的极简路线，**成本必须展示**。
- 概览页面**不强调投资回报率**。
- “每间可售房晚收入”需要展示，暂定为**次级结果位**。
- 敏感度分析页面的核心仍然是**敏感度分析本身**，当前分析逻辑整体保留。

### 视觉与品牌
- 本轮需要做**真正的视觉重构**。
- 可自由选择合适实现方式，核心目标是：
  1. 降低系统耦合
  2. 完成统一的视觉风格
  3. 完成统一的控件规范
- 项目收尾时，需要**把 Hopeful Finance 系统性改名**为目标命名（Rilo Analysis），包括：
  - 用户可见命名
  - 文档命名
  - 目录结构
  - 必要时内部结构命名

### 财务口径
- 总部费用（HQ Fee）：按 **总收入** 计提，不按医疗收入计提。
- Summary / 概览页现有一些结果参数是有价值的，**不要随意删掉**，而是整理与重排。

### 执行方式与工程原则
- 核心开发工具：**codex-cli**。
- 必须规避“改动只存在于未提交测试环境，最后丢失代码和上下文”的问题。
- 所有关键节点必须记录：
  - 收到问题
  - 开始解决
  - 遇到困难
  - 做出关键决策
  - 解决问题
- 任务需要尽量拆细。
- 默认按 **5 分钟节拍** 观测任务进度并记录。
- 持续观测 model usage；体感 API 变慢时及时做快照。

## 待继续确认（仅剩少量开放项）

- 工作台最终导航形态：当前策略为**保留三个页面，但做成统一工作台骨架**。
